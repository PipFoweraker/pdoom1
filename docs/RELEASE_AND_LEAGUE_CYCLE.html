P(Doom) Release & League Cycle

Internal runbook -- nomenclature, cadence, the fork rule, and the epoch-cut process. So future devs (and future Claude) do not mangle an epoch shift.

PINNED 2026-07-24  Reflective review due on/after 2026-08-24 (see issue in footer). Provisional until then -- especially the weekly-league cadence.

Read me first. There are two version numbers and two cadences. Keeping them apart is the whole point. If you change one thing before an epoch cut, let it be this: the ladder bump and the gameplay it scopes must ship in the same build.

1Nomenclature -- the six words

Use these exactly. The confusion this doc exists to kill is treating "version", "epoch", "league", and "board" as the same thing. They are not.

WordWhat it isWhere it livesChanges the board?
build_version Identifies the binary / .pck. Bumps every release and patch. Semver, e.g. 0.13.0. version.txt -> stamped into game_config.gd CURRENT_VERSION No. Cosmetic bumps must not fork the board.
ladder_version The ruleset. A small integer, rendered L2. Bumps only when gameplay / scoring / seed-generation rules change. ladder_version.txt -> stamped into game_config.gd LADDER_VERSION Yes. This is the board scope (get_board_version()).
epoch One value of ladder_version. "The game's current rules." Scores are comparable within an epoch, never across. = the current ladder_version Is the rotation.
league A promoted / featured seed on the current epoch. "This week's race." A fresh league = a new base seed, same rules. backend featured-board pointer + a promoted seed New board, same epoch.
seed The RNG root for one run/board. Anyone can share a seed; the promoted one is the league. get_weekly_seed() / manual promotion New seed = new board.
board The leaderboard for one (seed, ladder_version) pair. local JSON + PHP score API --

Board key = (seed, ladder_version). New seed rotates one board; a ladder bump rotates all boards.

2The two cadences

Two clocks, deliberately decoupled. This is the resolution of "how do we stay fresh without scattering scores."

ClockWhat movesCadenceForks the ladder?Ceremony
LEAGUE (the seed) A new promoted base seed. Same rules, fresh board -- "this week's ladder." ~weekly, manual at Pip's call for now NO -- same epoch Low. Post a seed, point the featured board at it.
GAME (the epoch) Balance patch + world-clock advance + any new gameplay. ladder_version bump. ~monthly YES -- new epoch High. This is the legible heartbeat: patch notes, meta rotation, argue about it.
Why this works A new seed does not change the rules, so weekly league rotation keeps competition fresh without making scores incomparable -- last week's board and this week's are both valid, just different races (chess.com-daily style). Only the monthly GAME change rotates the ruleset. Fragmentation was only ever a risk if rules forked weekly; seeds forking weekly is the intended freshness engine ("a league is just a promoted seed").

Rulings pinned 2026-07-24

  • Game changes monthly, league (seed) rotates weekly.
  • Rotation is manual (Pip's call), not clock-locked -- until we learn how long a league actually stays fresh. Do not hard-code a weekly timer yet.
  • World-clock patching is regular and predictable (advance the sim calendar ~a month, inject the month's real-world events). Because new events change the reachable game, this rides the monthly GAME change (it forks -- see section 3).
  • Weekly auto-rotation is blocked on a bug: get_weekly_seed() is frozen at weekly-2026-w0 (it reads ms-since-boot, not the date). While rotation is manual this does not bite; fix it before automating weekly promotion.

3The fork rule -- what starts a new epoch

The one question. Could two identical runs -- same seed, same player choices -- produce a different score, world trajectory, or RNG stream under this change?
YES → it forks. Bump ladder_version. Queue it for the next monthly epoch (do not ship on merge).
NO → it is cosmetic. Bump version.txt only. Ship any time. Everyone stays on the same board.
FORKS (epoch, monthly-batched) DOES NOT FORK (patch, any time)
Scoring rules (turn accrual, doom-integral tiebreak, thresholds) UI layout, panels, colours, fonts
Balance numbers that change outcomes (action costs, doom deltas, event probabilities, finance) Music, SFX, audio mix
Seed schedule / wave changes; advancing the world-clock; new events on existing seeds Art, icons, sprites, banners
RNG-stream changes (add/remove/reorder a draw, change turn order) -- breaks replays Copy / text / patch notes / onboarding wording
New content reachable in an existing seed's run; mortality-mechanism changes Bugfixes with NO behaviour change; tooling / CI / docs

Tie-break for the unsure author: if a saved replay from the previous epoch would fail to reproduce under your change, it forks -- by construction.

4Issue labels (derived from the fork rule)

The label is a function of the fork rule, not a vibe. Assign it by answering section 3.

LabelMeaningPR must...
league:<epoch> (e.g. league:v0.13) A forking change, queued for the next monthly epoch. touch ladder_version.txt (or be batched into the epoch commit that does).
patch:ui / patch:fix Non-forking, ships mid-week. NOT touch ladder_version.txt.
infra / build Non-forking, non-UI (packaging, telemetry, CI). NOT touch ladder_version.txt.
Known mislabels to fix (2026-07-24 audit) #789 is tagged both league:v0.13 AND ship:hotpatch-48h -- a forking change cannot be a non-forking hotpatch; keep the league tag, drop the hotpatch one. #788 (dev-badge = scoring integrity), #799 (telemetry), #805 (builds) are mislabelled patch:ui. #735/#700/#648 are scoring-critical but untiered. 5Epoch-cut runbook: L1 -> L2

Pinned numbering -- do not improvise L1 = the legacy v0.12.0 board (aliased, read-only -- what the friends-and-family cohort is playing now).
L2 = the v0.13 gameplay epoch (#789 hiring stitch + the build/ladder split live).
The one hazard that scatters scores NEVER cut a v0.13 gameplay build while ladder_version.txt still reads 1. It would key boards L1 and dump new-gameplay scores onto the aliased old-gameplay board -- the exact lie the split exists to prevent. The ladder bump and the #789 gameplay ride the SAME build.
  1. Prereq: PR #806 merged to main. That lands the split with ladder_version = 1, so the current live board is now epoch L1, and #789 gameplay is present in the tree.
  2. Choose the new base seed for L2's first league (manual promotion).
  3. One release commit bumps BOTH numbers. 
    # in version.txt:        0.12.0 -> 0.13.0
# in ladder_version.txt:      1 -> 2
python tools/sync_version.py          # stamps CURRENT_VERSION + LADDER_VERSION
    Atomic. Both bumps, one commit, same build as the #789 gameplay.
  4. Verify before building. 
    python tools/sync_version.py --check   # drift == fatal; must pass
# confirm GameConfig.get_board_version() now returns "L2"
  5. Build and smoke-test. 
    python tools/build_release.py          # nukes .godot, proves the freshness marker
    Run the exe once; play a turn; open the leaderboard and confirm the subtitle shows the L2 epoch.
  6. Backend (external, ssh to DreamCompute). 
    # make the legacy board readable under its new epoch name:
cp board_<seed>__v0.12.0.json  board_<seed>__L1.json
# confirm the score API accepts L2; point the featured board at (new seed, L2)
    Tracked in pdoom1-website #151.
  7. Publish and announce. 
    gh release create v0.13.0 --target main ...
    Then the friends / website / Rektango announcement. Kill stragglers on L1 mercilessly (they can hotpatch up).
Sanity check after the cut A player on the L2 build who dies posts to board (<new seed>, L2). A player still on the old v0.12 build posts to (<old seed>, L1). The two never mix. If a v0.13 score ever appears on an L1 board, step 3 was split across two builds -- stop and re-cut.

6Cheat card

Is my change cosmetic (UI/art/music/copy/no-behaviour fix)?
   YES -> bump version.txt only. Label patch:*. Ship mid-week. Same board.
   NO  -> it forks. bump ladder_version.txt too. Label league:<epoch>.
          Queue for the next MONTHLY epoch. Boards rotate.

Fresh competition without new rules?
   -> promote a new SEED (a new league). Same epoch, new board. Weekly-ish, manual.

Cutting the epoch?
   -> version.txt AND ladder_version.txt bump in ONE commit, same build as the gameplay.
      L1 = legacy v0.12 (read-only).  L2 = v0.13.
      sync_version.py --check must pass.  Alias old board.  Point featured board at (new seed, L2).
